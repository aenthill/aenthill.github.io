Introduction

Aenthill is a solution for allowing Docker containers to communicate with each other in order to fulfill a task in the current host directory.

Each of those Docker containers (called aent) is not aware of its peers. It is however able to receive and communicate events to them. In other words, an aent is not necessarily smart but it shares a part of the task to handle like an ant within its colony.

Installation

Aenthill is distributed in a binary form and the only requirement is to have Docker installed and running on your host (always use the latest version).

In order to get aenthill, run the following command:

$ curl -sf https://raw.githubusercontent.com/aenthill/aenthill/master/install.sh | BINDIR=/usr/local/bin sh

This will install the latest release of Aenthill in the /usr/local/bin directory.

You may of course change the destination directory by updating BINDIR value.

If you want to install a specific version of Aenthill, you may also run:

$ curl -sf https://raw.githubusercontent.com/aenthill/aenthill/master/install.sh | BINDIR=/usr/local/bin sh -s version

You may find all available versions in the releases page.

The binary aenthill also let you update to the latest version with the following command:

$ aenthill upgrade [-t version]

Last but not least, verify installation with:

$ aenthill --version

Commands

Aenthill provides the aenthill command-line program, which contains many useful commands.

init

Aenthill relies on an aenthill.json file (called manifest) which gathers all aents of your ecosystem.

You may create this file in your current directory by running:

$ aenthill init

add

Once your manifest created, you may add aents by using:

$ aenthill add image [image...]

Aenthill will add them to your manifest and send the event ADD to them.

rm

You may also remove aents by using:

$ aenthill rm image [image...]

Aenthill will remove them from your manifest and send the event REMOVE to them.

How does it work?

As explained before, an ecosystem of aents is working by sending events to each other.

An event may be sent by Aenthill directly or by another aent.

Let’s start with Aenthill:

$ aenthill add aenthill/cassandra

This command will send the event ADD to the aent aenthill/cassandra by running:

$ docker run -ti --rm
-v "/var/run/docker.sock:/var/run/docker.sock"
-v "/host/project/dir:/aenthill"
-e "PHEROMONE_FROM="
-e "PHEROMONE_WHOAMI=aenthill/cassandra"
-e "PHEROMONE_HOST_PROJECT_DIR=/host/project/dir"
-e "PHEROMONE_CONTAINER_PROJECT_DIR=/aenthill"
-e "PHEROMONE_LOG_LEVEL=INFO"
aenthill/cassandra aent "ADD"

There are several things to note:

  • Aenthill starts the aent aenthill/cassandra by calling its binary aent with the event name as first argument
  • It mounts the current host working directory /host/project/dir to the aent directory /aenthill
  • It binds the host Docker socket
  • It populates some environment variables

Those environment variables allows the aent to understand within which context it has been awaken.

PHEROMONE_FROM

The event sender (empty if sended by Aenthill).

PHEROMONE_WHOAMI

The event recipient (the aent itself).

PHEROMONE_HOST_PROJECT_DIR

The host project directory, useful if the aent sends an event to another aent.

PHEROMONE_CONTAINER_PROJECT_DIR

The path of the project directory in the aent.

PHEROMONE_LOG_LEVEL

The log level as defined by the user with Aenthill.

aent

As you understand, an aent is a Docker image which has a binary called aent. This binary should accept two arguments:

  • The event name (a string)
  • The payload (a string too) which contains the data associated to the event

The last argument has to be optional (for instance, it is never filled by Aenthill).

The entire process will stop if the binary aent returns an exit code different than 0.

You may now wondering how an aent is able to communicate with another aent.

Actually, it works the same as Aenthill: use the Docker client binary with almost the same options. Just don’t forget to:

  • Set PHEROMONE_FROM with your aent image name
  • Set PHEROMONE_WHOAMI with the recipient image name

Too much complicated? Don’t worry, we provide Hermes, a tool for sending events inside a Docker container.

Hermes

Hermes is a tool which can be used directly inside an aent to send events.

Its goal is to ease the communication of your aent with others aents of your ecosystem.

Installation

Below an example of Dockerfile:

FROM alpine:3.7

[...]

# Required for Hermes to know which interpreter to use when calling the Docker client binary.
ENV SHELL "/bin/sh"

# Installs Docker client.
ENV DOCKER_VERSION "18.03.1-ce"
RUN wget -qO- https://download.docker.com/linux/static/stable/x86_64/docker-$DOCKER_VERSION.tgz | tar xvz -C . &&\
    mv ./docker/docker /usr/bin &&\
    rm -rf ./docker

# Installs Hermes.
ENV HERMES_VERSION "latest version"
RUN wget -qO- https://github.com/aenthill/hermes/releases/download/$HERMES_VERSION/hermes_linux_amd64.tar.gz | tar xvz -C . &&\
    mv ./hermes /usr/bin &&\
    rm -f LICENSE README.md

[...]

Commands

dispatch

$ hermes dispatch event [payload]

Hermes will send the given event and payload to all aents in the manifest which handles it.

reply

$ hermes reply event [payload]

Hermes will send the given event and payload back to the aent which which has awaken yours.

It uses the environment variable PHEROMONE_FROM to know which aent to reply to.

Commands

set:handled-events

$ hercule set:handled-events [event...]

This command allows your aent to specify which events it handles by updating the manifest.

Next time Hermes will dispatch an event, your aent will be awaken only if it handles it.

If no event given, your aent will always be awaken by incoming events!

This command should be called when your aent is awaken by the ADD event from Aenthill
